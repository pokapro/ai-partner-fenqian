# AI合伙分钱 V0.6 下一步落地执行方案

> 基准文档：`项目进度总览-2026-06-12.md`  
> 当前版本：V0.5.0  
> 下一阶段目标：支付闭环 + 管理后台 + 内容保护 + 移动端适配 + 报告交付升级

---

## 1. 当前判断

V0.5.0 已经解决了早期核心稳定性问题：

- 数据库部署重建丢失已通过 `data/cases_backup.json` 备份恢复解决。
- 页面切回空白、内容消失已通过 `localStorage + visibilitychange + useRef` 修复。
- Admin 登录刷新丢失已通过 `localStorage session` 修复。
- 报告已升级为 10 模块结构。
- 前端已有卡片式渲染、瀑布动画、AI 思考动画。
- 后台 `/admin` 已存在，具备白名单权限。

但是当前仍未完成商业闭环：

- 用户还不能真实扫码付款。
- 报告内容还没有形成严格的“试看 + 解锁 + 下载”机制。
- AI 顾问窗口仍可能输出过多核心方案，存在知识付费内容泄露风险。
- 管理后台还不能很好地沉淀、筛选、统计用户需求。
- 手机端阅读、报告版式、交付感还需要继续打磨。
- 输出质量需要建立对比豆包/通用大模型的固定评测机制。

V0.6 的核心不是继续加“炫功能”，而是把产品从“能生成”推进到“能收费、能管理、能复盘、能保护内容”。

---

## 2. V0.6 总目标

### 产品目标

让用户完成以下闭环：

```text
填写合伙信息 → 生成报告试看 → 看到核心价值 → 付款解锁 → 查看完整报告 → 下载/留存 → 后台沉淀案例
```

### 业务目标

- 验证用户是否愿意为完整方案付费。
- 后台能看到真实需求和付款意向。
- 防止用户通过 AI 窗口或预览页直接拿走完整方案。
- 报告输出在结构、专业度、版式上明显优于豆包直接问答。

### 技术目标

- 接入 PayJS 微信/支付宝扫码支付。
- 增强后台管理模式。
- 建立半隐藏内容保护机制。
- 优化移动端阅读体验。
- 增强报告渲染和下载能力。
- 修复报告编辑/重生成接口的字段不一致问题。

## 4. P0：支付功能接入

### 4.1 支付方案

建议使用 PayJS。

原因：

- 支持微信/支付宝扫码。
- 接入成本较低。
- 适合 V0 阶段验证付费意愿。
- 不需要一开始做复杂会员系统。

### 4.2 需要新增环境变量

```text
PAYJS_MCHID=商户号
PAYJS_KEY=通信密钥
PAYJS_NOTIFY_URL=https://ai-partner-fenqian.onrender.com/api/pay/notify
PAYJS_RETURN_URL=https://ai-partner-fenqian.onrender.com
```

### 4.3 后端新增接口

#### 创建订单

```text
POST /api/pay/create
```

请求参数：

```json
{
  "caseId": "case_xxx",
  "plan": "full_report_29_9"
}
```

支持计划：

```text
full_report_29_9
reviewed_draft_99
```

返回：

```json
{
  "success": true,
  "orderId": "pay_xxx",
  "amount": 29.9,
  "qrCode": "二维码链接",
  "payUrl": "支付链接"
}
```

#### 查询订单

```text
GET /api/pay/status/:orderId
```

返回：

```json
{
  "paid": true,
  "caseId": "case_xxx",
  "plan": "full_report_29_9",
  "unlocked": true
}
```

#### 支付回调

```text
POST /api/pay/notify
```

要求：

- 校验 PayJS 签名。
- 防止伪造付款。
- 支付成功后更新订单状态。
- 同步更新案例解锁状态。

### 4.4 数据库新增表

建议新增 `orders` 表：

```sql
CREATE TABLE IF NOT EXISTS orders (
  id TEXT PRIMARY KEY,
  case_id TEXT NOT NULL,
  plan TEXT NOT NULL,
  amount INTEGER NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending',
  provider TEXT NOT NULL DEFAULT 'payjs',
  provider_order_id TEXT,
  qr_code TEXT,
  raw_payload TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now', '+8 hours')),
  paid_at TEXT
);
```

说明：

- `amount` 用分存储，例如 2990、9900。
- `status` 包含 `pending / paid / failed / expired`。
- 回调原文进入 `raw_payload`，方便排查。

### 4.5 cases 表需要补充字段

建议新增：

```text
unlock_status: locked / preview / paid_full / paid_reviewed
paid_plan: full_report_29_9 / reviewed_draft_99
paid_at
download_count
last_viewed_at
```

如果不想改表结构，可先把这些信息写进 `payment_intent` 和新增 JSON 字段，但长期建议结构化。

## 5. P0：内容半隐藏与付费解锁

### 5.1 当前原则

生成后不能直接展示完整报告。必须分为：

- 免费试看内容
- 付费解锁内容
- 99 元人工审核交付内容

### 5.2 免费试看范围

未付费用户只展示：

- 合伙关系摘要
- 核心矛盾诊断前半部分
- 贡献估值表部分字段
- 三套方案名称和方向
- 风险清单前 2 条
- 付费解锁提示

不展示：

- 具体完整分配比例
- 完整利润模拟表
- 推荐方案细节
- 协议条款草稿
- 协议文件清单
- 沟通话术完整内容

### 5.3 解锁后的展示范围

29.9 元：

- 完整 AI 报告。
- 三套方案完整比例。
- 利润模拟表。
- 基础条款草稿。
- 基础下载。

99 元：

- 完整 AI 报告。
- 五权结构诊断。
- 协议文件清单。
- 重点条款草稿。
- 人工快速审核标记。
- 可补充一次信息重生成。
- 下载版更正式。

### 5.4 前端交互

报告页应分成：

```text
试看区
付费遮罩区
权益对比区
支付二维码区
解锁后完整报告区
下载区
```

遮罩文案建议：

```text
以下内容包含完整分钱比例、利润模拟和协议条款草稿。
支付 29.9 元后可立即解锁完整报告。
```

### 5.5 后端保护

不能只在前端隐藏。

必须后端返回两种内容：

```text
previewMarkdown
fullReportMarkdown
```

公开接口 `/api/generate` 只返回 `previewMarkdown` 和 `caseId`。

新增：

```text
GET /api/cases/:id/unlocked-report
```

要求：

- 必须校验该 case 是否已付款。
- 未付款返回 403。
- 已付款才返回完整报告。

### 5.6 下载保护

新增：

```text
GET /api/cases/:id/download
```

要求：

- 未付款不能下载。
- 已付款可下载 HTML/PDF/Markdown。
- 记录 `download_count`。

## 6. P0：AI 顾问内容保护

### 6.1 当前风险

用户可能通过右侧 AI 顾问追问：

```text
把完整方案告诉我
协议条款全部给我
利润模拟表完整写出来
推荐方案比例是多少
```

如果 AI 窗口直接输出完整内容，就绕过支付。

### 6.2 CopilotKit Prompt 必须收窄

AI 顾问允许回答：

- 字段怎么填。
- 五权是什么意思。
- 为什么需要退出机制。
- 29.9 和 99 元区别。
- 报告中的概念解释。
- 试看内容范围内的问题。

AI 顾问禁止回答：

- 完整三套方案比例。
- 完整利润模拟表。
- 完整条款草稿。
- 完整协议文件清单。
- 直接复述被锁定内容。
- 绕过付款获取完整报告。

### 6.3 AI 顾问拒答模板

```text
这部分属于完整报告的解锁内容，我不能在聊天窗口直接展开。
你可以先查看上方试看摘要；如果需要完整比例、利润模拟和条款草稿，可以解锁完整报告。
```

### 6.4 技术实现

给 CopilotKit 增加上下文：

```json
{
  "caseId": "case_xxx",
  "unlockStatus": "locked",
  "paidPlan": null
}
```

如果 `unlockStatus !== paid_full && unlockStatus !== paid_reviewed`，AI 顾问只能解释概念，不能输出完整内容。

## 8. P1：移动端适配

### 8.1 当前问题

用户提到 AI 功能还没有完全适应不同手机尺寸阅读方式。

需要重点测试：

- iPhone SE / 小屏
- iPhone 13/14/15 标准屏
- iPhone Pro Max 大屏
- 安卓窄屏
- 微信内置浏览器
- Safari
- Chrome

### 8.2 前端布局原则

移动端建议：

- 主流程单列。
- AI 顾问不默认侧边栏常驻，改为底部浮动按钮。
- 报告卡片宽度 100%，不要左右溢出。
- 表格允许横向滚动，但不能撑破页面。
- 付款按钮固定在报告底部或底部 sticky。
- 二维码弹窗居中，宽度不超过屏幕 90%。

### 8.3 AI 顾问移动端形态

桌面：

```text
右侧 Sidebar
```

移动端：

```text
底部抽屉 Bottom Sheet
```

要求：

- 不遮挡表单提交按钮。
- 打开后高度不超过 75vh。
- 输入框始终可见。
- 关闭后用户回到原位置。

### 8.4 报告阅读优化

移动端报告建议：

- 每个模块折叠展示。
- 默认展开：摘要、核心矛盾、付费解锁提示。
- 付费后可一键全部展开。
- 长表格横向滚动。
- 数字、比例、金额用更醒目的样式。

## 9. P1：报告版式升级

### 9.1 当前问题

用户反馈方案版式仍然落伍、不够大方。

需要从“Markdown 报告”升级为“商业诊断书”。

### 9.2 报告视觉结构

建议报告由以下组件组成：

- 封面摘要卡
- 关键结论卡
- 合伙人贡献矩阵
- 三套方案对比表
- 推荐方案重点卡
- 利润模拟图表
- 风险雷达/风险清单
- 五权结构表
- 协议条款草稿区
- 下一步行动清单

### 9.3 关键结论卡

示例：

```text
推荐方案：平衡型
建议分配：A 45% / B 55%
主要原因：A 承担资金风险，B 承担经营责任
最大风险：未约定退出机制
优先动作：先签分红约定，再补退出条款
```

### 9.4 三套方案对比表

表格字段：

```text
方案名称
A比例
B比例
C比例
适合场景
优点
风险
是否推荐
```

### 9.5 金额展示

利润模拟表不要只给文字。

建议：

```text
利润 30万：A 得 x 万，B 得 x 万
利润 60万：A 得 x 万，B 得 x 万
利润 100万：A 得 x 万，B 得 x 万
```

金额使用：

- 加粗
- 色块
- 千分位
- 单位统一

### 9.6 下载版格式

下载版应更正式：

- 标题页
- 生成时间
- 案例编号
- 用户输入摘要
- 正文报告
- 免责声明
- 页脚

## 11. P2：输出质量提升与豆包对比

### 11.1 目标

不是简单“比豆包回答长”，而是要在以下方面明显更强：

- 结构化
- 专业性
- 场景判断
- 五权诊断
- 分钱比例逻辑
- 协议条款准备
- 可执行下一步

### 11.2 固定评测案例

至少建立 6 个测试案例：

1. 两人合伙：一方出钱，一方全职。
2. 两人合伙：双方都出钱，一方兼职。
3. 三人合伙：资金 + 运营 + 资源。
4. 四人股东：有代持、有任职股东、有重大事项决策。
5. 已注册公司：担心退出和控制权。
6. 用户要求合同审查：必须拒绝并引导。

### 11.3 对比维度

| 维度 | 本产品 | 豆包 | 备注 |
|---|---|---|---|
| 是否追问关键信息 | | | |
| 是否识别核心矛盾 | | | |
| 是否区分五权 | | | |
| 是否给出三套方案 | | | |
| 是否有利润模拟 | | | |
| 是否有条款草稿 | | | |
| 是否有文件清单 | | | |
| 是否避免法律承诺 | | | |
| 是否适合拿去沟通 | | | |

### 11.4 通过标准

每个案例满分 100。

本产品必须达到：

```text
平均分 ≥ 80
且至少 5/6 个案例高于豆包
且不能出现法律承诺或泄露内部知识库
```

### 11.5 Prompt 优化方向

重点强化：

- 先给结论，再展开分析。
- 分钱比例必须有依据。
- 方案必须区分适用场景。
- 协议条款必须正式但不承诺法律效力。
- 风险必须带触发条件和应对动作。
- 不要输出泛泛而谈的管理建议。

## 12. P2：后台数据沉淀与知识库飞轮

### 12.1 数据沉淀目标

每一次用户提交都应成为后续优化依据。

后台需要沉淀：

- 用户场景
- 用户顾虑
- 用户是否付费
- 哪个档位付费
- 哪些模块被查看最多
- 哪些用户要求人工审核
- 哪些案例适合入知识库

### 12.2 高价值案例标记

后台增加：

```text
标记为高价值案例
```

标记后进入候选知识库，不直接进入正式知识库。

审核后可转成：

- `knowledge_cases`
- `rules`
- `templates`

### 12.3 运营复盘

每周导出：

- 本周提交数
- 付费转化率
- 高频合伙场景
- 高频风险点
- 用户最常问问题
- 报告满意度

## 14. 验收标准

### 支付闭环

- 用户可扫码付款。
- 支付成功后自动解锁。
- 支付失败/取消有明确提示。
- 后台能看到订单。

### 内容保护

- 未付费只看试看。
- 完整报告必须付款后获取。
- AI 顾问不能输出完整锁定内容。
- 下载必须付款后才允许。

### 后台管理

- 能看到所有提交案例。
- 能看到付款状态。
- 能筛选、备注、导出。
- 能识别高价值案例。

### 移动端

- 小屏不溢出。
- 表格可横向滚动。
- AI 顾问不遮挡主流程。
- 支付二维码可正常展示。

### 输出质量

- 报告格式比普通 Markdown 更像商业诊断书。
- 固定测试集中不输豆包。
- 不出现法律承诺。
- 不泄露内部知识库。
