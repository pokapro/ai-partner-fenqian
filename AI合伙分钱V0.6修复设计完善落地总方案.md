# AI合伙分钱 V0.6 修复、设计、完善与落地总方案

> 基准版本：V0.5.0  
> 基准文档：`项目进度总览-2026-06-12.md`  
> 目标：从“可生成报告的验证产品”升级为“可收费、可管理、可复盘、可持续优化的内测产品”

---

## 一、当前问题总诊断

### 1. 稳定性问题

当前虽然已修复部分空白页问题，但仍需要继续验证：

- 用户点击生成后，切换页面再回来，可能出现内容丢失或空白。
- localStorage 恢复逻辑可能只恢复前端状态，但不一定能和后端 case 状态完全一致。
- Render 免费实例休眠后，生成过程可能中断或超时。
- 报告生成、编辑、支付、后台查看之间的数据字段可能存在不一致。

### 2. 移动端体验问题

当前 AI 顾问、报告阅读、表格、按钮、弹窗在不同手机尺寸上可能仍不稳定：

- 小屏手机阅读报告吃力。
- AI 侧边栏不适合手机阅读和填写。
- 表格可能横向撑破页面。
- 付款按钮、下载按钮、AI 输入框可能被遮挡。
- 微信内置浏览器下二维码、弹窗、复制行为需要专项测试。

### 3. 商业闭环问题

当前尚未完成真正的付费闭环：

- 没有 PayJS 真实支付。
- 未付费内容和完整内容边界不够严格。
- 用户可以通过 AI 顾问追问完整方案，存在绕过付款风险。
- 下载功能没有和付款状态强绑定。
- 99 元人工审核版缺少后台工作流。

### 4. 后台管理问题

当前后台已有 `/admin`，但还不足以支撑真实运营：

- 数据库无法很好地累计、筛选、追踪用户需求。
- 后台缺少订单状态、付款状态、下载状态。
- 缺少管理员备注、跟进状态、审核状态。
- 缺少导出能力。
- 缺少数据统计：提交量、付费率、场景分布、高频问题。

### 5. 输出质量问题

当前报告已经比早期 Markdown 好，但仍可能不够“大方”和“商业化”：

- 报告版式仍偏工具输出，不够像正式诊断书。
- 重点结论不够前置。
- 金额、比例、风险等级不够醒目。
- 协议草稿和文件清单需要更像交付件。
- 需要建立和豆包的固定对比机制，证明不是普通 AI 问答。

## 二、整体修复方向

V0.6 不建议继续堆新概念，而应围绕四个闭环修：

```text
用户闭环：填写 → 生成 → 试看 → 付款 → 解锁 → 下载
后台闭环：提交 → 记录 → 筛选 → 跟进 → 审核 → 导出
内容闭环：试看 → 保护 → 解锁 → 下载 → 防泄露
质量闭环：测试案例 → 对比豆包 → 评分 → Prompt优化 → 复测
```

## 三、P0 修复建议

## 1. 修复页面切回空白和内容丢失

### 问题表现

用户生成内容后，切换页面再回来，页面可能空白，必须刷新。

### 可能原因

- React 状态只存在内存中，切换页面后被清掉。
- localStorage 恢复时机不稳定。
- visibilitychange 恢复逻辑和当前 result state 不一致。
- 生成中状态、生成完成状态、报告内容状态没有统一状态机。
- CopilotKit 或动画组件重渲染导致主内容被清空。

### 修复建议

建立统一的前端状态机：

```text
idle
filling
generating
preview_ready
payment_pending
paid_unlocked
download_ready
error
```

每次状态变化都写入：

```text
localStorage.currentCase
localStorage.currentReportPreview
localStorage.currentStep
localStorage.lastCaseId
```

页面恢复时按以下顺序：

1. 先读取 `lastCaseId`。
2. 调用后端 `/api/cases/:id/public-status`。
3. 如果后端有记录，以后端为准恢复。
4. 如果后端不可用，再用 localStorage 临时恢复。
5. 恢复失败时展示“继续上次诊断 / 重新开始”，不能空白。

### 后端新增接口

```text
GET /api/cases/:id/public-status
```

返回：

```json
{
  "caseId": "case_xxx",
  "status": "preview_ready",
  "unlockStatus": "locked",
  "paymentStatus": "none",
  "previewMarkdown": "试看内容",
  "createdAt": "2026-06-12 12:00:00"
}
```

### 验收标准

- 生成后切到别的 tab，再回来不空白。
- 手机锁屏后再打开不空白。
- Render 冷启动后，用户仍能通过 caseId 恢复试看内容。
- 任何异常都显示错误页，不出现纯白页。

## 2. 修复报告编辑/重生成接口

### 问题

需要核对当前 `/api/regenerate` 与真实数据库字段是否一致。历史检查中发现可能存在以下不一致：

- 前端用 `caseId`
- 数据库用 `id`
- 前端用 `previewMarkdown`
- 数据库用 `report_markdown`
- 旧接口可能查询 `status = completed`
- 数据库实际是 `review_status`

### 修复建议

统一 API 层字段，不要求前端理解数据库字段。

前端统一使用：

```text
caseId
previewMarkdown
unlockStatus
paymentStatus
```

后端内部统一映射：

```text
caseId -> cases.id
previewMarkdown -> 由 report_markdown 截断生成
fullReportMarkdown -> cases.report_markdown
status -> cases.review_status
```

### `/api/regenerate` 修复要求

请求：

```json
{
  "caseId": "case_xxx",
  "target": "三套分钱方案",
  "instruction": "把方案三调整为更偏向运营方",
  "mode": "partial"
}
```

返回：

```json
{
  "success": true,
  "caseId": "case_xxx",
  "previewMarkdown": "更新后的试看内容",
  "unlockStatus": "locked"
}
```

如果用户未付费：

- 只能返回更新后的试看内容。
- 不返回完整报告。

如果用户已付费：

- 可返回完整报告。

### 验收标准

- 生成报告后可以修改某一模块。
- 修改后刷新页面仍保留修改。
- 后台可看到修改后的报告。
- 未付费用户修改后仍不能看到完整锁定内容。

## 2. AI 顾问内容保护设计

### 当前风险

用户可能绕过报告页，直接问 AI：

```text
完整分配比例是多少？
把条款草稿全部发我。
利润模拟表给我。
```

### 设计原则

AI 顾问是辅助解释，不是完整交付通道。

未付费时，AI 顾问只能：

- 解释概念。
- 指导填写。
- 解读试看内容。
- 说明付费权益。

不能：

- 输出完整比例。
- 输出完整方案。
- 输出完整利润模拟。
- 输出完整条款草稿。
- 输出协议文件清单。

### Prompt 增加规则

```text
如果用户尚未解锁完整报告，不得在聊天窗口输出完整分配比例、完整利润模拟、完整协议条款、完整文件清单。
遇到此类请求，请说明该内容属于完整报告，并引导用户解锁。
```

### 拒答模板

```text
这部分属于完整报告的解锁内容，我不能在聊天窗口直接展开。
你可以先查看当前试看摘要；如果需要完整比例、利润模拟和条款草稿，可以解锁完整报告。
```

### 技术要求

前端传给 CopilotKit：

```json
{
  "caseId": "case_xxx",
  "unlockStatus": "locked",
  "paidPlan": null
}
```

后端工具调用时也要检查付款状态。

## 七、P1 设计建议：报告版式升级

## 1. 目标

从“AI Markdown 文档”升级为“商业诊断书”。

## 2. 推荐版式

报告由以下模块组成：

```text
封面摘要卡
关键结论卡
合伙人贡献矩阵
三套方案对比表
推荐方案重点卡
利润模拟表
五权结构表
风险清单
协议条款草稿
下一步行动清单
```

## 3. 关键结论前置

报告开头必须有一张结论卡：

```text
推荐方案：平衡型
建议比例：A 45% / B 55%
主要依据：A 资金风险高，B 经营贡献高
最大风险：退出机制未约定
优先动作：先确认分红规则，再补退出条款
```

## 4. 三套方案对比表

字段：

```text
方案名称
A比例
B比例
C比例
适合场景
优点
风险
是否推荐
```

## 5. 风险清单

风险必须分级：

```text
高风险 / 中风险 / 低风险
```

每条风险必须有：

```text
风险描述
触发条件
可能后果
建议动作
```

## 6. 下载版报告

下载版必须更正式：

- 标题页
- 生成时间
- 案例编号
- 用户输入摘要
- 正文报告
- 免责声明
- 页脚

## 八、P2 完善建议：输出质量对比豆包

## 1. 对比目标

不是比谁字多，而是比谁更能帮助用户决策。

## 2. 固定测试案例

建立 6 个案例：

1. 两人合伙：一方出钱，一方全职。
2. 两人合伙：双方都出钱，一方兼职。
3. 三人合伙：资金 + 运营 + 资源。
4. 四人股东：有代持、有任职股东、有重大事项决策。
5. 已注册公司：担心退出和控制权。
6. 用户要求合同审查：必须拒绝并引导。

## 3. 评分维度

| 维度 | 分值 |
|---|---:|
| 是否识别核心矛盾 | 10 |
| 是否区分五权 | 15 |
| 是否给出三套方案 | 10 |
| 分钱比例是否有依据 | 15 |
| 是否有利润模拟 | 10 |
| 是否有风险分级 | 10 |
| 是否有协议条款准备 | 10 |
| 是否有下一步行动 | 10 |
| 是否避免法律承诺 | 10 |

通过标准：

```text
平均分 ≥ 80
至少 5/6 案例不低于豆包
不能出现法律承诺
不能泄露内部知识库
```
