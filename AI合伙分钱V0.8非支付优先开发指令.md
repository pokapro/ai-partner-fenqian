# AI 合伙分钱 V0.8 非支付优先开发指令

> 生成日期：2026-06-15  
> 执行原则：支付功能暂缓。本阶段不做真实支付、不做扫码收款、不做复杂会员系统。先完善数据库、访问效果、阅读体验、互动体验和后台运营闭环。

---

## 一、本阶段目标

把当前 V0.7 系统升级为可以稳定内测的 V0.8-beta。

用户侧要达到：

- 能顺利填写信息。
- 能稳定生成报告。
- 刷新、切页面、返回后内容不丢。
- 手机端阅读舒服。
- AI 互动像“合伙分钱顾问”，不是普通聊天窗口。
- 未开通支付前，可以通过“申请完整报告 / 联系顾问 / 管理员解锁”完成转化验证。

后台侧要达到：

- 每一次提交都能沉淀到数据库。
- 管理员能查看、筛选、备注、跟进用户需求。
- 优质案例可以脱敏沉淀进案例库。
- 后台能支持后续人工审核和销售转化。

---

## 二、明确不做的内容

本阶段暂不做：

- 真实微信支付、支付宝支付。
- PayJS 正式联调。
- 自动开票。
- 复杂会员中心。
- 多级代理分销。
- 在线合同签署。

但需要先预留：

- `unlocked` 状态。
- 管理员手动解锁。
- 完整报告访问接口。
- 未解锁预览报告接口。

原因：即使支付没开，内容也不能直接全量暴露，否则无法验证后续付费。

## 三、最高优先级任务

### 1. 数据库和后台运营闭环

当前系统已经有数据库基础，但还不够像“运营后台”。下一步要把用户提交变成可跟进的商机记录。

#### 需要补齐的数据字段

在 `cases` 表中确认或新增以下字段：

| 字段 | 说明 |
|---|---|
| `case_id` | 案例编号 |
| `created_at` | 提交时间 |
| `contact` | 联系方式 |
| `scene_mode` | 小店/标准/公司化 |
| `partner_count` | 合伙人数 |
| `annual_profit` | 预期年利润 |
| `input_summary` | 用户输入摘要 |
| `preview_markdown` | 预览版报告 |
| `report_markdown` | 完整版报告 |
| `unlock_status` | 未解锁/管理员解锁/测试白名单 |
| `review_status` | 待审核/已审核/需补充/无效 |
| `followup_status` | 新提交/已联系/已发报告/待补充/已转化/无效 |
| `admin_note` | 管理员备注 |
| `quality_score` | 人工打分 |
| `promoted_to_knowledge` | 是否已沉淀到知识库 |

#### 后台页面需要支持

- 查看所有用户提交。
- 按提交时间排序。
- 按跟进状态筛选。
- 按审核状态筛选。
- 查看用户原始输入。
- 查看预览报告和完整报告。
- 给用户记录备注。
- 修改跟进状态。
- 管理员手动解锁完整报告。
- 一键导出 CSV。
- 一键将优质案例脱敏升为知识库案例。

#### 必修 Bug

- 修复后台 CSV 导出中 `today` 未定义问题。
- 修复备份恢复接口中 `database`、`persist()` 作用域错误。
- 检查 `admin_note`、`followup_status` 字段是否真实存在，不存在就做迁移。
- 白名单接口不要把明文密码返回给前端。

### 2. 访问效果和页面稳定性

目标：用户不要因为页面切换、刷新、手机后台切换导致报告消失。

#### 用户侧状态设计

必须明确以下状态：

| 状态 | 含义 | 页面表现 |
|---|---|---|
| `idle` | 未开始 | 显示表单 |
| `generating` | 正在生成 | 显示进度和“可稍后回来查看” |
| `preview_ready` | 预览已生成 | 显示预览报告 |
| `manual_review` | 等待人工审核 | 显示已提交提示 |
| `unlocked` | 已解锁完整报告 | 显示完整报告和下载 |
| `error` | 生成失败 | 显示重试和联系入口 |

#### 恢复策略

- 生成案例后立即保存 `caseId`。
- localStorage 只保存：`caseId`、状态、预览内容、生成时间，不保存完整报告。
- 页面刷新时优先用 `caseId` 调后端恢复状态。
- 如果后端还在生成，继续展示进度。
- 如果后端已完成，恢复预览报告。
- 如果报告修改成功，要重新切分章节并重新渲染，不允许显示旧内容。

#### 必修 Bug

- 修复报告封面乱码：`分���方案`。
- 修复 `small_biz` 场景显示不正确。
- 修复版本号不统一问题。
- 修复生成后返回页面可能空白的问题。

### 3. 报告阅读体验

当前报告最大问题是“内容有了，但交付感不够”。要从 AI 文本升级为商业诊断书。

#### 报告结构

固定为 12 个模块：

1. 执行摘要
2. 合伙结构快照
3. 核心矛盾诊断
4. 贡献估值表
5. 五权结构诊断
6. 三套分钱方案对比
7. 利润模拟表
8. 推荐方案与调整条件
9. 风险清单
10. 协议条款草稿
11. 沟通话术与下一步行动
12. 免责声明

#### 阅读体验要求

- 第一屏必须能看到推荐结论。
- 不要让用户先看大段解释。
- 三套方案必须表格化。
- 利润模拟必须金额清楚。
- 风险清单必须分级。
- 合同条款必须像合同，不像建议。
- 手机端表格要变成卡片，不强迫横向拖动。
- 去掉过多 emoji，改为专业标签。
- 减少多色渐变，改成统一的商业报告风格。

#### 建议视觉风格

- 主色：深蓝灰、墨绿、浅灰。
- 背景：浅灰页面 + 白色报告纸张。
- 风险：用小标签，不用整块大红色。
- 表格：浅灰表头、细边框、数字右对齐。
- 条款：使用正式协议编号样式。
- 圆角：控制在 8px 左右。

---

### 4. 手机端体验

手机端不是简单压缩桌面版，而是要单独优化。

#### 手机端布局

- 表单单列。
- 分组折叠：基础信息、贡献信息、风险信息、联系方式。
- 报告正文一列阅读。
- 目录可折叠。
- 底部固定操作条只保留 2-3 个核心按钮。

#### AI 顾问窗口

手机端不要用桌面侧边栏。

建议改为：

- 右下角悬浮按钮：`问顾问`
- 点击后打开底部抽屉。
- 抽屉高度 85%-90%。
- 可关闭，不遮挡报告。
- AI 回答不得直接输出完整付费报告，只能解释、追问、引导补充信息。

#### 验收尺寸

必须测试：

- 360px 宽 Android
- 375px 宽 iPhone SE
- 390px 宽 iPhone 14/15
- 412px 宽 Android 大屏
- 1440px 桌面

---

### 5. 互动体验

用户不是只要“生成一篇报告”，而是需要被引导把复杂合伙情况说清楚。

#### 表单互动

- 增加字段解释，但不要写太长。
- 用户填写不完整时，给出温和提示。
- 出资、出力、资源、风险这些字段要有示例。
- 年利润输入支持区间和单个数字。
- 合伙人信息建议用卡片形式，不要一堆输入框挤在一起。

#### AI 一键填表

要求：

- 用户用一句话描述后，AI 自动填表。
- 自动填完后要让用户确认。
- 不能直接提交，必须用户确认。
- 不确定的信息标记为“需补充”。

#### 报告修改

要求：

- 支持局部修改：只改方案、只改条款、只改话术。
- 修改后刷新当前报告预览。
- 后台保留修改历史。
- 修改次数可记录，用于判断用户需求强度。

---

### 6. AI 输出质量

目标：对比豆包时不输，最好明显更像专业交付。

#### Prompt 优化方向

- 结论先行。
- 明确推荐一个主方案。
- 三套方案必须有差异。
- 每套方案必须写适用条件和风险。
- 五权结构必须完整。
- 条款草稿必须合同化。
- 不允许只写泛泛建议。
- 不允许只说“建议协商”。

#### 标准测试案例

至少准备 6 个测试案例：

| 案例 | 场景 |
|---|---|
| A | 一方出钱，一方全职运营 |
| B | 两人都出钱，但一方资源更强 |
| C | 三人平均股权但贡献不均 |
| D | 品牌/IP/渠道资源入股 |
| E | 小店亲友合伙 |
| F | 公司化项目，涉及控制权和退出 |

每个案例用本系统和豆包各生成一份，按以下维度评分：

- 结构完整度
- 方案可执行性
- 权责专业度
- 条款格式
- 风险识别
- 阅读体验

只有本系统明显更强，才适合对外宣传“专业合伙分钱报告”。

## 五、验收标准

### 用户访问效果

- [ ] 手机和电脑都能正常打开。
- [ ] 生成后切页面回来不空白。
- [ ] 刷新后能恢复案例。
- [ ] 生成失败有重试入口。
- [ ] 报告第一屏能看懂结论。

- [ ] 每次提交都入库。
- [ ] 后台能看到完整用户需求。
- [ ] 后台能备注和修改跟进状态。
- [ ] CSV 能导出。
- [ ] 优质案例能升库。

### 阅读体验

- [ ] 报告不像普通聊天回答。
- [ ] 三套方案清楚。
- [ ] 利润模拟清楚。
- [ ] 五权诊断清楚。
- [ ] 合同条款有正式格式。
- [ ] 手机端不需要来回横拖。

### 互动体验

- [ ] AI 一键填表可用。
- [ ] AI 顾问不遮挡阅读。
- [ ] AI 顾问不会直接泄露完整报告。

## 六、给 ClawX 的执行口径

支付功能先不要碰。本轮只做 V0.8-beta 的非支付完善：数据库沉淀、后台跟进、访问稳定、报告阅读体验、手机端适配、AI 互动体验和输出质量。核心目标是让用户可以顺利提交真实合伙信息，系统能稳定生成可读的预览报告，后台能沉淀和跟进用户需求，完整报告通过管理员手动解锁或人工发送完成内测转化验证。
